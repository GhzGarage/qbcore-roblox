qb-core Roblox Port

A Roblox/Luau port of the core QBCore flow: session-locked DataStore profiles, multi-character select and spawn, server-authoritative inventory and status decay, native admin tooling, synced weather, and a full set of QBCore-style native UIs — no NUI, no web views.

Overview

What this project is, and what it deliberately is not.

This is a Rojo-managed Roblox place that re-implements the core loop of the FiveM QBCore framework in Luau. The design philosophy carries over: the server owns all persistent gameplay data — clients only display state and request changes through remotes. Player.UserId replaces FiveM license identifiers, and a single session-locked DataStore profile per Roblox account holds all of that account's characters.

Accounts & Characters

Profile loading with DataStore session locking, character select / create / delete / spawn, up to 5 slots per account.

Persistence

Money, banking statements, job, crew, charinfo, position, metadata, inventory items, and saved appearance survive across sessions.

Status & Medical

Server-side hunger/thirst decay, HUD bars, armor absorption, death screen with timed self-respawn, EMS revives.

Inventory & Shops

30-slot weight-based inventory, five-slot hotbar, two-pane external inventory UI, and proximity item shops with stock and quantity purchases.

Banking

Personal and society accounts, bank cards, PIN-gated ATMs, queued citizen-ID transfers, limits, and persistent statements.

Management

Standalone job/crew boss panel with indexed rosters, nearby hiring, grades, removal, offline queues, and shared funds.

Weapons

Inventory-backed Roblox Tool equip flow on the endorsed Weapons Kit, with ammo item consumption and persisted loaded rounds.

World

Server-authoritative day/night clock, synced weather presets with rain/snow/thunder, blackout, and client-side time-of-day grading.

Admin

Graded permissions (user/mod/admin/god), a native admin panel, bans and whitelist DataStores, and slash commands.

Social

Saved HumanoidDescription appearance, categorized clothing/barber shops, outfit share codes, emotes, and proximity stage music.

Scope note

This documents the framework code under src/. The repository also contains a separate island map generation pipeline, which is out of scope for these docs.

Getting Started

Serve the project into Studio with Rojo and enable DataStore access.

Rojo workflow

Use serve, not build, for normal Studio development:

.\serve-rojo.bat

Then connect the Roblox Studio Rojo plugin to:

localhost:34872

Keep the terminal open while you work — file edits in the workspace stream into Studio live. Use rojo build only when you want a standalone place file: 

rojo build default.project.json --output qbcore-roblox.rbxlx

Studio setup

DataStores do not work in Studio Play mode by default. Enable:

Game Settings → Security → Enable Studio Access to API Services
Duplicate objects

If Studio starts showing duplicate QBCore/QBShared objects: stop Play mode, disconnect the Rojo plugin, run tools/clean-rojo-studio-objects.lua in the Command Bar, then reconnect Rojo.

Treat the workspace as the source of truth for every Rojo-managed object (ReplicatedStorage/QBShared, ServerScriptService/QBCore, all QB* LocalScripts). ServerScriptService preserves unknown children so imported systems like WeaponsSystem can live next to the managed QBCore folder.

First playtest

  1. Serve with Rojo and press Play in Studio — Studio testers are automatically god rank.
  2. Create a character on the character select screen (first/last name).
  3. The appearance editor opens automatically on first spawn.
  4. Try Tab for inventory, /commands to list available commands, /admin for the admin panel.

Naming convention

<Name>.server.lua becomes a Script, <Name>.client.lua a LocalScript, and plain <Name>.lua a ModuleScript. The suffix is how Rojo picks the instance class, so it can never be dropped.

Architecture

How a request flows from a client action to persisted data.

Everything follows the same server-authoritative loop. A client UI (inventory, admin panel, character select) invokes a RemoteFunction or fires a RemoteEvent defined in QBRemoteNames.lua. The server entrypoint Main.server.lua routes it to the owning service, which validates permissions and input, mutates the in-memory PlayerData through PlayerClass / InventoryService, and pushes the delta back down with player:UpdateClient(key, value) → the PlayerDataUpdated event. 

-- lifecycle at a glance
Players.PlayerAdded
  → Access.CheckJoin()              -- closed / banned / whitelist gate
  → PlayerService.OnPlayerJoin()    -- claim Account_<UserId> session lock
  → client: character select UI
  → Remotes.SelectCharacter
  → PlayerService.SelectCharacter() -- build PlayerClass, spawn, fire PlayerLoaded
  → status loop ticks               -- position capture + hunger/thirst decay
  → PlayerRemoving / BindToClose    -- Save() + profile:Release()

Persistence is one DataStore key per Roblox account (QBCore_PlayerAccounts → Account_<UserId>) holding { characters = {[citizenId] = data}, nextCid }, session-locked via ProfileStore with a 35-second lock timeout and 60-second autosave. A separate QBCore_CitizenIndex store maps citizen IDs back to UserIds for offline lookups, and bans/whitelist each live in their own DataStore.

Server Services

Every module under ServerScriptService/QBCore. Booted in order by Main.server.lua, which also wires all remotes and sets Players.CharacterAutoLoads = false.

server PlayerService.lua Account/character lifecycle, spawn resolution, status loop, world environment.

The heart of the framework — claims the account profile on join, exposes character CRUD, spawns characters, and runs the periodic status loop.

  • PlayerService.Players / PlayerService.PlayersByCitizenIdLive maps of online PlayerClass instances.
  • GetPlayer(userId) · GetPlayerByCitizenId(citizenId) · GetPlayers()Lookup helpers for online players.
  • GetPlayersByJob(job, checkOnDuty) → list, countFilter online players by job, optionally only on-duty.
  • GetOfflinePlayerByCitizenId(citizenId) → PlayerClass?Read-only offline instance via the citizen index + lock-free peek.
  • GetAccountUserIdByCitizenId(citizenId) → UserId?Read-only citizen-index lookup used for validated queued transfers and text filtering.
  • StartStatusLoop()Every Config.StatusInterval seconds: capture position, decay hunger/thirst per Config.StatusDecay.
  • GetCharacterList(player)Returns {citizenId, cid, firstname, lastname, job, cash, bank}[] for the select screen.
  • SelectCharacter(player, citizenId) → ok, err?Builds the PlayerClass, spawns the character, fires PlayerLoaded; opens the appearance editor on first spawn.
  • CreateCharacter(player, firstname, lastname) → citizenId, err?Enforces slot limit and 20-char names, generates a citizen ID, seeds starter items.
  • DeleteCharacter(player, citizenId) → ok, err?
  • RespawnPlayer(player, playerObj, spawnCFrame?, health?) → ok, err?
  • ApplyWorldEnvironment()Applies Config.World Lighting/Sky/Clouds/post-effects.
  • Logout(player) · SaveAllAndRelease()Logout currently saves and kicks (in-place character switching is on the roadmap).
server BankingService.lua Personal/society accounts, cards, ATMs, queued transfers, and statements.

Inspired by qb-banking and adapted to Roblox ProximityPrompts. Every request revalidates the exact bank or ATM location, account permission, card/PIN, and available funds on the server.

  • BankingService.Start()Creates invisible prompt anchors in Workspace.QBBankingLocations and binds banking remotes.
  • GetBanking(access)Returns accessible personal/society accounts, ATM usage, limits, location, and newest-first statements.
  • BankingAction("deposit" | "withdraw" | "transfer" | "order_card", payload)Validates proximity, account access, whole-dollar limits, funds, and ATM credentials.
  • WithdrawSocietyFunds(job, amount, playerObj)Atomic provider used by optional society-backed paychecks.
  • GetOrganizationFunds(type, name) Â· ChangeOrganizationFunds(type, name, delta, statement)Atomic job/crew shared-account API used by ManagementService.
  • DeliverPendingTransfers(player, playerObj)Credits ready offline transfers once, saves their processed IDs, then clears the inbox.

Personal statements and ATM counters live under PlayerData.banking. Society accounts and transfer inboxes use dedicated DataStores; transfer IDs prevent duplicate delivery after a retry or crash.

server ManagementService.lua Job/crew rosters, nearby hiring, grades, removal, offline queues, shared funds.

Inspired by qb-management, with crew data replacing gang references and a standalone native GUI replacing qb-menu. Every request revalidates the prompt location, current organization, boss flag, grade ceiling, and target distance on the server.

  • ManagementService.Start(BankingService, AppearanceService)Creates invisible prompt anchors in Workspace.QBManagementLocations and binds management and wardrobe remotes.
  • GetManagement(access)Returns the authorized organization, grades, indexed online/offline roster, nearby candidates, cash, and shared balance.
  • ManagementAction("hire" | "set_grade" | "fire" | "deposit" | "withdraw", payload)Performs the validated mutation and returns a refreshed snapshot.
  • OnCharacterLoaded(player, playerObj)Applies safe offline changes and synchronizes the character into job and crew roster indexes.

Because account profiles are session locked, changes to offline or cross-server members are queued in a dedicated DataStore and applied on their next character load. Existing characters appear in indexed rosters after their first post-install load. The panel's Wardrobe shortcut opens the clothing-only saved-outfit manager after boss access is revalidated; shared stashes remain dependent on the future container system.

server ProfileStore.lua Self-contained session-locked DataStore wrapper (ProfileService-style).

Replaces oxmysql from the FiveM original. Claims an exclusive session lock per key so two servers can never write the same account at once.

  • ProfileStore.new(datastoreName) → store
  • store:StartSessionAsync(key, defaultData) → profile, err?35s lock timeout, up to 5 claim retries at 6s intervals, 60s autosave.
  • store:PeekAsync(key) → dataLock-free read for offline lookups.
  • profile:Save() · profile:Release() · profile:Reconcile(defaults)
  • ProfileStore.Reconcile(data, defaults) · ProfileStore.DeepCopy(value)Exported helpers used across services.
server InventoryService.lua Server-authoritative inventory core: slots, weight, move/give/use, external providers.

Slots 1..Config.Inventory.Slots (default 30); the first HotbarSlots (5) are the hotbar. Items are validated against QBShared.Items on every load.

  • GetSnapshot(playerObj) → {items, slots, hotbarSlots, maxWeight, totalWeight}Hydrated client-facing view.
  • RegisterExternalProvider(kind, provider) · GetOpenSnapshot(..., access?) · HandleExternalAction(...)Shared two-pane route used by shops and ready for container providers.
  • CanAddItem(playerObj, itemName, amount, slot?, info?) → ok, err?Dry-run weight/slot check.
  • AddItem(playerObj, itemName, amount, slot?, info?, reason?) · RemoveItem(...)
  • MoveItem(playerObj, fromSlot, toSlot) → ok, err?Merges compatible stacks.
  • GiveSlot(playerObj, slot) → ok, err?Gives 1 unit to the closest player within Config.Inventory.GiveDistance, rolling back on failure.
  • UseSlot(playerObj, slot) → ok, err?Dispatches to a registered handler or applies consume hunger/thirst deltas; removes the item unless removeOnUse == false.
  • CreateUseableItem(itemName, handler) → okRegisters a custom use-handler — used by MedicalService and WeaponService.
  • SetItemData(playerObj, itemName, key, value, slot?)Mutates item.info[key] (e.g. persisted weapon ammo).
  • HasItem / GetItemBySlot / GetItemByName / GetItemsByName / GetItemCount / SetInventory
  • SeedStarterItems(playerData) · ReconcilePlayerData(playerData) · GetTotalWeight(items) · GetFreeWeight(playerObj)
server MedicalService.lua Death, respawn, armor absorption, and medical item handlers.
  • SetArmor(playerObj, amount) → okClamps 0–100; sets metadata plus a Humanoid Armor attribute.
  • RevivePlayer(targetObj, targetPlayer, reviveCFrame, reviveHealth) → ok, err?
  • RequestRespawn(player) → ok, err?Enforces Config.Medical.DeathScreen.RespawnDelay; optionally wipes inventory.
  • ApplyWeaponDamage(system, target, amount, ...)Armor absorption before Humanoid:TakeDamage; wired as the Weapons Kit damage callback.
  • Start(InventoryService)Registers useable handlers for every item with a medical table: "heal" (bandage), "armor", and "revive" (firstaid — requires an on-duty ambulance job and proximity).
server WeaponService.lua Inventory-backed Tool equip flow on the imported Weapons Kit.

Registers every item with type == "weapon" as a useable item — using it equips/holsters the Tool. Templates live in ServerStorage/QBWeaponTools.

  • WeaponService.Start(InventoryService)Also auto-collects loose imported Tools and WeaponsSystem folders from Workspace/StarterPack.
  • GetToolFolder()

On equip, the template is cloned, weapon.config values are applied onto its Configuration (creating missing ValueBase objects), and the kit's CurrentAmmo is tracked so reloads are charged against the player's ammo item (weapon.ammoItem) — the loaded count persists in item.info.ammo.

server AdminService.lua Permission-checked backend for the native admin panel.

Minimum rank to open the panel: admin. Keeps an in-memory ring buffer of the last 100 admin actions.

  • GetContext(player) → {allowed, rank, server, players, items, jobs, crews, vehicles, weather, logs}One snapshot powering every panel page.
  • HandleAction(player, action, payload) → ok, resultDispatch table, permission-checked per call.

Actions: selfHeal, announce, setTime, toggleFreezeTime, setWeather, toggleFreezeWeather, toggleBlackout, gotoPlayer, bringPlayer, teleportToCoords, healPlayer, kickPlayer, banPlayer, addMoney, setMoney, giveItem, spawnVehicle, setJob, setCrew.

server Access.lua Join gate: graded permissions, bans, whitelist.
  • GetRank(userId) → rankName, rankLevelGame creator and Studio testers are auto-god; otherwise the highest matching entry in Config.Server.Permissions.
  • HasPermission(userId, minRank) → boolean · IsAdmin(userId)
  • BanPlayer(userId, reason, expireAt) · UnbanPlayer(userId) · IsBanned(userId) → isBanned, reasonQBCore_Bans DataStore; expired bans auto-lift.
  • WhitelistPlayer(userId) · UnwhitelistPlayer(userId) · IsWhitelisted(userId)QBCore_Whitelist DataStore; admins always pass.
  • CheckJoin(userId) → ok, kickReason?Checks Config.Server.Closed, bans, then whitelist.
server CommandService.lua & Commands.lua TextChatCommand registry plus the default command set.

Replaces FiveM's "/" chat scan with Roblox's TextChatService command layer. All feedback goes through toast notifications, not chat lines.

  • CommandService.Add(name, help, arguments, argsRequired, callback, permission?, secondaryAlias?)arguments = {{name, help}, ...} builds the usage string; callback gets (player, args, rawText); permission defaults to "user".
  • CommandService.GetVisibleCommands(userId) → info[]Filtered by rank — powers /commands.
  • CommandService.ListAll registered commands with usage strings.

See the full command table.

server AppearanceService.lua Saved HumanoidDescription appearance backend (qb-clothing style).

Serialized appearance shape: { scales, bodyParts, clothing, colors, accessories }, stored on PlayerData.appearance and re-applied on every respawn. Categorized shops create invisible ProximityPrompt anchors under Workspace.QBClothingShops; category limits are enforced again on the server.

  • Serialize(humanoidDescription) → table · Deserialize(table) → HumanoidDescription
  • ApplyToCharacter(character, appearance) → ok
  • Start(PlayerService)Creates clothing-shop prompts and binds the saved-outfit/share-code API.
  • OpenEditor(player, playerObj, isNewCharacter, context?)Opens a full editor, categorized shop, outfit-only room, or validated management wardrobe.
  • Preview(player, payload)Sanitizes and live-applies a preview.
  • SaveAppearance(player, playerObj, payload) → ok, err?Validates catalog ownership via MarketplaceService:PlayerOwnsAsset when Config.Appearance.ValidateOwnership is on.
  • CancelEdit(player, playerObj)
  • DeleteOutfitCodes(playerData)Revokes a character's global codes during character deletion.

Saved outfits contain only classic/layered clothing and wearable accessories. Hair, faces, makeup/future identity fields, body parts, scales, and skin colors are preserved on the recipient and never stored in a share code.

server TimeSyncService.lua Server-authoritative day/night clock.

Advances Lighting.ClockTime once per second (auto-replicates to clients). A full day takes Config.World.Time.DayLengthMinutes (default 48 real minutes).

  • SetTime(hour, minute?) → ok · GetTime() → number
  • SetFreeze(state) · IsFrozen() → boolean
server WeatherService.lua Synced weather cycling, transitions, and blackout.

Publishes state as attributes on a ReplicatedStorage/QBWeatherState folder and fires WeatherStateUpdated. Auto-cycles with weighted randomness every 10–20 minutes by default.

  • GetState() → snapshot{currentWeather, nextWeather, transitionStartedAt, transitionDuration, frozen, dynamic, blackout, ...}
  • SetWeather(name, transitionSeconds?) → ok, message
  • SetFrozen(frozen) · SetDynamic(dynamic)
  • SetBlackout(enabled)Toggles all Light instances under CollectionService-tagged instances (Config.Weather.BlackoutLightTags).
server VehicleService.lua Vehicle template spawning and cleanup.

Templates live in ServerStorage/QBVehicleModels; spawned instances go under Workspace/SpawnedVehicles and are stamped with QBVehicle* attributes (name, label, plate, spawn ID, owner UserId).

  • GetVehicleDefinition(name) → definition, err?Resolves by name / label / modelName / alias, case-insensitive.
  • SpawnVehicle(player, name, options?) → instance, definition, plateOptions include {plate, cframe, distance, attributes, parent, anchored, canCollide, disableScripts}.

server VehicleShopService.lua Showroom displays, persistent purchases, financing, and timed test drives.

Roblox adaptation of qb-vehicleshop. It creates anchored showroom clones and invisible ProximityPrompt anchors, then revalidates location, catalog eligibility, price, ownership, payment, and exit occupancy for every action.

  • GetVehicleShop(access)Returns the filtered catalog, template availability, owned vehicles, finance balances, and shop limits.
  • VehicleShopAction("purchase" | "finance" | "test_drive" | "spawn_owned" | "finance_payment", payload)
  • Start(VehicleService)Builds four configured displays plus showroom and finance prompts under Workspace.QBVehicleShop.

server GarageService.lua Public garage ownership checks, storage, condition persistence, and retrieval.

Roblox adaptation of the public flow from qb-garages. Vehicle state follows the familiar convention: 0 out, 1 garaged, and 2 impounded.

  • GetGarage(access)Returns vehicles assigned to the current garage, state, plate, finance balance, stored condition, and nearby deposit eligibility.
  • GarageAction("retrieve" | "store", payload)Revalidates proximity, ownership ID, garage assignment, template availability, persistence, and spawn occupancy.
  • Start(VehicleService)Creates configured invisible ProximityPrompt anchors under Workspace.QBGarages.

The two initial public locations are placeholders at Vector3.new(0, 0, 0). Depot, job, and house restrictions remain future work.

The OOP wrapper around one character's PlayerData — the object you get from PlayerService.GetPlayer(userId). This is the primary API for gameplay code.

playerObj:AddMoney("cash", 250, "delivery-payout")
playerObj:Notify("You earned $250", "success", 4000)

Offline instances

Player.new(source, ...) with source == nil marks an offline/admin-tool instance (Offline = true) — network calls become no-ops, so the same methods work for offline edits.

The per-character PlayerData shape, reconciled onto every load from Config.Player.CharacterDefaults.

  name = "<DisplayName>",           -- refreshed on select

  charinfo = {
    firstname = "Firstname", lastname = "Lastname",
    birthdate = "00-00-0000", gender = 0, nationality = "USA",
  },
  job = {
    name = "unemployed", label = "Civilian", onduty = false, type = "none",
    grade = { name = "Freelancer", level = 0, payment = 10, isboss = false },
  },
  crew = {
    name = "none", label = "No Crew",
    grade = { name = "none", level = 0, isboss = false },

Client Scripts

Every LocalScript under StarterPlayerScripts. All UI is native Roblox UI — ScreenGuis built in Luau, no web views.

client QBCoreClient

Character select / create / delete UI. Card list with Play/Delete (confirm step) plus a create panel. Destroys itself once PlayerLoaded fires. Responsive with compact/tiny breakpoints.

client QBHUD

Compact HUD: name, job + duty, cash and bank, and status bars for HP / AR / FOOD / WATR / STRS. Also renders the ammo panel for the equipped weapon. Disables the core health GUI and reset button.

client QBInventory

30-slot inventory + 5-slot hotbar. Personal access stays single-pane; shops/external providers add a responsive second pane with product price, stock, quantity, and Buy controls.

client QBNotify

Toast stack (max 5 visible) fed by the Notify remote. Types: success, error, warning, info, primary. Durations auto-convert ms→s, clamped 1.5–12s.

client QBMenu

Reusable QBCore-style list menu used by emotes and stage music. Exposed via _G.QBMenu and PlayerGui bindables. See QBMenu.

client QBAdmin

Native admin panel: Dashboard, Players, Items, Jobs, Crews, Vehicles, Logs, Environment, Developer pages (Reports / Chat / Leaderboard are placeholders). Opened via /admin.

client QBAmbulance

Death screen with countdown; the Respawn button unlocks after RespawnDelay and calls RequestRespawn. Keyboard E / gamepad X.

client QBAppearance

In-game avatar editor: hair, hats, glasses, faces, classic + layered clothing, shoes, accessories, skin tones, and 5 body-scale sliders. Lists owned catalog items via AvatarEditorService, previews live (0.15s debounce).

client QBEmotes

Emote menu (/emotes) via QBMenu — default emotes plus any owned emotes on the applied HumanoidDescription. Re-selecting a playing emote stops it.

client QBStageMusic

Stage speaker menu via QBMenu: fixed tracks + live Creator Store search results, play/stop/volume controls sent over StageMusicControl.

client QBTimeCycle

Per-frame time-of-day color grading — smoothstep-samples Config.World.TimeCycle.Keyframes by clock time, blends the active weather layer, and drives Lighting, ColorCorrection, Atmosphere, SunRays, Bloom, and clouds.

client QBWeather

Local precipitation and lightning: a camera-tracking ParticleEmitter configured from the sampled weather state, plus lightning flash and optional thunder audio.

client QBWeaponAmmo

Client-side mirror of the server ammo clamp — prevents kit-predicted refills beyond actual reserve and toasts "No X left" on empty reload.

QBMenu

The reusable native menu controller — the Roblox equivalent of qb-menu.

Exposed two ways: _G.QBMenu (OpenMenu, RefreshMenu, CloseMenu, IsOpen, plus Selected / Closed BindableEvents) and equivalently via BindableFunctions/Events under PlayerGui.QBMenu. Esc or gamepad B closes. 

_G.QBMenu.OpenMenu({
  { header = "Emotes", isMenuHeader = true },
  {
    header = "Wave",
    txt = "Give a friendly wave",
    params = {
      event = "emote:play",          -- or action/callback
      args = { emote = "wave" },
      -- isServer = true, target/remote for server-bound actions
    },
  },
})

Item fields: header, txt, isMenuHeader, disabled, hidden, icon/image, shouldClose, and params = {event, args, action/callback, target/remote, isServer}.

Vehicles

A ~30-entry catalog in QBShared/Vehicles.lua, built with two factory helpers.

  • vehicle(def) — drivable cars. Defaults: brand = "Roblox", fuel = 100, gloveboxSlots = 5, gloveboxWeight = 10000. An attributeProfile (one of emergency, sedan, sports, supercar, offroad, utility, pickup, suv, van) expands into a full physics-tuning attributes table.
  • board(def) — personal transport (skateboards, hoverboards): fuel = 0, trunkSlots = 0.

Each entry carries name, label, price, modelName (the Studio template to match in ServerStorage/QBVehicleModels), category, trunkSlots/trunkWeight, aliases[], and an optional Toolbox assetId. Spawn with /car <vehicle> or the admin panel; delete the closest with /dv. The dealership exposes every catalog entry except taxi and police; prices currently default to $0.

Templates required

Catalog entries only spawn if a matching model exists in ServerStorage/QBVehicleModels. Installing a template for every entry is a roadmap item.

Weapon Setup

Weapon item data lives in QBShared/Items.lua; the actual Roblox Tool templates go in ServerStorage/QBWeaponTools, driven by the endorsed Weapons Kit. 

ServerScriptService
  WeaponsSystem          -- imported Weapons Kit
ServerStorage
  QBWeaponTools
    Pistol               -- Tool template, matched by weapon.toolName

Each weapon item can override the Tool template's Configuration values:

weapon = {
    toolName = "Pistol",
    ammoItem = "pistol_ammo",
    config = {
        AmmoCapacity = 12,
        FireMode = "Semiautomatic",
        HitDamage = 24,
        MaxSpread = 2,
        ShotCooldown = 0.18,
    },
}

Existing ValueBase objects are reused, missing number/string/boolean values are created, and omitted values keep the template's settings. Other kit values work too: GravityFactor, MuzzleFlashSize0/1, RecoilDecay, RecoilMin/Max, TotalRecoilMax, ShotEffect, CasingEffect.

Importing the endorsed pistol

  1. Insert the free Roblox pistol from Toolbox — click No on the StarterPack prompt.
  2. Move the Pistol Tool into ServerStorage/QBWeaponTools and the imported WeaponsSystem folder into ServerScriptService. (The server also tries to auto-collect loose copies at runtime.)
  3. Test with /giveitem <userId-or-name> weapon_pistol 1.
Kit patches

Studio Command Bar fixes live in tools/: setup-weapons-kit.lua (create folders), patch-weapons-shoulder-camera.lua (read-only C0 error spam), and patch-weapons-local-crosshair.lua (remote players hijacking the local crosshair).

Weather & Time

Eight presets in QBShared/Weather.lua, blended server-side and rendered locally.

Preset order: CLEAR, EXTRASUNNY, CLOUDS, OVERCAST, FOG, RAIN, THUNDER, SNOW. Each defines label, random-cycle weight, wind, clouds, lighting, atmosphere, color correction, sun rays, precipitation (kind/rate/speed), and thunder timing.

  • Server: WeatherService picks the next preset by weighted random every 10–20 minutes and blends over TransitionSeconds (45s default).
  • Client visuals: QBWeather renders precipitation particles and lightning; QBTimeCycle blends the weather layer into the time-of-day grade every frame.
  • Blackout: /blackout disables every Light under instances tagged QBBlackoutLight or StreetLight.
  • Clock: TimeSyncService drives Lighting.ClockTime; a full day lasts 48 real minutes by default. Control via /time and /freezetime.

Admins can drive all of this from the Environment tab of the admin panel too.

Roadmap

From TODO.md — working resource slices exist, but this is not a full QBCore ecosystem yet. Suggested milestone order: production hardening → inventory economy → vehicles → jobs → world.

Battle-tested profile library (or hardened retries/telemetry), Luau static checks and Studio smoke tests, structured audit logs, schema-migration helpers, localization.

True in-session character switching (logout currently kicks), richer creation fields (birthdate, gender, nationality, phone), spawn selection, deeper deletion cleanup.

Shop selling and persistent stock, world drops, stashes/trunks/gloveboxes, crafting, item metadata (durability and serials), invoices, loans, and management stashes.

Map duty prompts, blips, and grade-gated actions; police/EMS/mechanic/taxi/delivery jobs; dispatch, fines, warrants, evidence; crew progression.

Templates for every catalog entry, impound/depot and persistent world parking, job/house garages, keys/locks, real fuel/repair integrations, and trunk/glovebox inventories.

Tool templates for every configured weapon, licenses and job restrictions, weapon/ammo shops, downed state (beyond death screen), combat audit tooling.
